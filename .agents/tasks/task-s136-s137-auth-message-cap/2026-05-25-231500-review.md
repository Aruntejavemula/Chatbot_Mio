# Email auth endpoints and free-plan message cap

This change adds email/password authentication (signup, login, forgot-password, reset-password, resend-verification) to the FastAPI backend and the corresponding Flutter screens. It also introduces a daily message counting system for free-plan users capped at 20 messages/day. The backend uses placeholder Supabase calls and in-memory storage for both rate limiting and message counting. The frontend implements the full auth flow with form validation, navigation, and a progress-bar widget that shows message usage in the sidebar.

**Watch for:** The signup flow saves an access token and sets the current user before email verification is confirmed (confirmed). The message count widget is hardcoded to `count: 0` and will never reflect real usage without a backend integration (confirmed). The in-memory rate limiter in `auth.py` duplicates the existing `rate_limit.py` middleware without reusing it (confirmed). The reset-password token is passed as a URL query parameter, which will appear in server logs and browser history (likely).

## High-level view

The backend auth router implements five endpoints behind a custom IP-based rate limiter. This is distinct from the existing user-keyed `rate_limit.py` middleware because auth endpoints serve unauthenticated users, but the two implementations share no code and will diverge over time. All endpoints return placeholder data — no actual Supabase integration exists yet.

The message guard service uses an in-memory dict keyed by `msg_count:{user_id}:{date}`. It has no eviction mechanism; old date keys accumulate until the process restarts. The chat router gates message sending on this service and exposes a count query endpoint at `GET /chat/tokens/message-count` (note: prefixed under `/chat`).

The frontend auth screens follow established patterns — `ConsumerStatefulWidget`, form validation, DioException handling, dark/light theming. The signup flow navigates to a verify-email screen but also persists the access token and sets `currentUser` immediately, meaning a user who hasn't verified their email will appear "logged in" until the next session.

The `MessageCountWidget` in the sidebar is wired with a static `count: 0` and gated behind `isProPlan = true` (hardcoded), so it will never render in the current state. The widget itself is ready to accept real data once a provider feeds it.

## Signup saves state before email verification

In `auth_repository.dart`, `signUpWithEmail` persists the access token via `saveTokenSecurely` and sets `currentUserProvider` to the returned user. The backend responds with `email_verified: false`, but the repository doesn't check that flag — the user is treated identically to a verified login. The screen then navigates to `VerifyEmailScreen`, but if the user restarts the app, the persisted token will attempt to restore a session with an unverified account. Whether the backend eventually rejects unverified tokens depends on the real Supabase integration, but the frontend's auth state doesn't model the distinction.

```dart
final token = response['access_token'] as String;
await _authService.saveTokenSecurely(token);
final user = UserModel.fromJson(response['user'] as Map<String, dynamic>);
_ref.read(currentUserProvider.notifier).state = user;
```

The `signInWithEmail` path has the same structure, but the backend login endpoint already blocks unverified emails with a 403. The asymmetry matters: signup stores the token unconditionally.

## Duplicate rate-limiting infrastructure

The existing `backend/app/middleware/rate_limit.py` provides a `RateLimiter` class with a `reset()` method and proper key eviction. The new `auth.py` creates `_AuthRateLimiter` with an almost-identical sliding-window algorithm but:

- No `reset()` method (untestable in isolation)
- No eviction of empty keys after timestamps expire — only filters timestamps, never deletes empty lists
- A separate global instance that won't be touched by any test teardown using the existing rate limiter's `reset()`

The functional reason for a separate implementation is that auth endpoints are unauthenticated (keyed on IP rather than user_id). But the `RateLimiter` class itself is key-agnostic — it accepts any string key. A factory function that takes a key-extraction strategy (IP vs user_id) and shares the storage class would avoid this duplication.

## MessageGuard memory growth

`MessageGuard._counts` is a plain dict. Keys are of the form `msg_count:usr123:2026-05-25`. A new key is created for every user every day. There is no background task, no LRU eviction, no TTL check. The production Redis equivalent would use `EXPIRE`, but the in-memory stand-in does not simulate that behavior. For development and single-instance use this is tolerable, but it should be documented as a known limitation or addressed with a daily purge.

```python
def _get_key(self, user_id: str) -> str:
    today = date.today().isoformat()
    return f"msg_count:{user_id}:{today}"
```

## Token in URL query parameters

The router passes the reset token as a query parameter:

```dart
GoRoute(
  path: AppRoutes.resetPassword,
  pageBuilder: (context, state) {
    final token = state.uri.queryParameters['token'] ?? '';
    ...
  },
),
```

On mobile (deep links), the URL will appear in intent logs. On web, it will be in the address bar, browser history, and potentially logged by intermediary proxies. This is a common pattern (Supabase itself uses URL-based magic links), but the token carries password-reset authority. If the app ever runs in a web context, this should be paired with single-use token enforcement on the backend and short expiration windows.

## Message count widget is dead code today

The widget is gated by:

```dart
final bool isProPlan = true;
// ...
if (!isProPlan) ...[
  MessageCountWidget(count: 0, limit: AppConstants.freeMessageLimit),
]
```

`isProPlan` is always `true`, so the condition is always false. Even if toggled, `count` is hardcoded to 0 — there's no provider or API call feeding real data. The widget implementation itself (color thresholds, progress bar) is ready, but wiring is missing.

## Plan field absent from auth middleware

`get_current_user` in `middleware/auth.py` returns `{"user_id": token[:8]}`. The chat router does:

```python
plan = current_user.get("plan", "free")
```

This will always resolve to `"free"`. When the real JWT decoding is implemented, the plan field must be included in the decoded payload or fetched from a user store. Until then, no user can bypass the message limit — including actual pro users — because the middleware doesn't surface their plan.
