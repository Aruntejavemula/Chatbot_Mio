# File upload, artifact viewer, and prompt maker for chat input

Three UI features land in a single batch: a file-upload bottom sheet with preview bar (S36), a full-screen artifact viewer for code/HTML/markdown in AI responses (S37), and a sparkle-button prompt improver wired to a backend endpoint (S38). All three integrate into the existing chat screen's input bar and bubble widgets. The approach is widget-per-feature with callbacks threading state up to `_ChatScreenState`.

**Watch for:** The prompt maker instantiates a fresh `ChatService()` on every tap, bypassing any shared Dio instance or auth interceptor lifecycle (confirmed). Selected files are tracked in local state but never sent with the message (confirmed). The artifact type detector uses naive substring matching that will false-positive on conversations *about* code fences (likely).

## High-level view

The file upload widget renders a `+` icon that opens a modal bottom sheet with a 2x2 grid (Photo, Camera, File, Scan). Files chosen through platform pickers are wrapped in `SelectedFileInfo` and surfaced to the parent via callback. The parent renders a preview bar above the input, but the `_sendMessage` method clears the file list without ever attaching files to the request payload, so the feature is UI-only with no backend integration.

The artifact viewer detects artifact type by scanning message content for markdown code fences or header counts, then shows a "View" / "Preview" button below AI bubbles. Tapping opens a full-screen modal with a code view (line-numbered), a markdown renderer, or an HTML placeholder. The detection logic is string-contains-based and doesn't account for messages that merely discuss code fences without containing actual artifacts.

The prompt maker places a sparkle (`auto_awesome`) icon between the `+` button and the text field. When tapped with text present, it calls `POST /chat/make-prompt` and replaces the input with the server's response. Each invocation creates a new `ChatService()` instance rather than reusing the one available through the app's DI (Riverpod providers).

<details>
<summary>Issues (6)</summary>

1. **Files never sent with message** — `_sendMessage` clears `_selectedFiles` but never passes them to the backend. Either wire file paths into the chat payload or document this as intentionally UI-only.
2. **New ChatService per prompt-maker tap** — `PromptMakerWidget._onTap` does `ChatService().makePrompt(...)`, creating a fresh Dio instance each time. Use a shared instance (from provider or constructor injection) to avoid redundant interceptor setup and potential auth-token race conditions.
3. **Artifact detection false positives** — `detectArtifactType` uses `content.contains('```dart')` etc., so any message *discussing* code fences (e.g. "use \`\`\`dart to format your code") triggers the artifact button. Consider requiring the fence to start at the beginning of a line (`\n\`\`\`dart\n`).
4. **No file-size limit enforcement** — `_pickFile` and `_pickPhoto` accept arbitrarily large files. There's no client-side size guard, so users can select multi-GB files that will either crash image rendering or fail silently on upload.
5. **Double-tap on prompt maker fires parallel requests** — `_isLoading` prevents the UI from showing the button, but there's a timing gap between the tap and the first `setState` where a second tap can slip through. A simple early-return check on `_isLoading` at the top of `_onTap` would close it.
6. **`onFileRemoved` callback is dead code** — `FileUploadWidget` accepts `onFileRemoved` but never invokes it internally. The actual removal is handled by the preview bar in `chat_screen.dart`. Remove the unused parameter or document why it exists.

## File upload: UI complete, backend wiring absent

The disconnect is in `_sendMessage`:

```dart
setState(() {
  _hasText = false;
  _selectedFiles = [];
});
```

The files vanish without being attached to any request. The existing `streamMessage` API in `ChatService` takes a `content` string and model/provider params but no file references. Users will select files, see them in the preview bar, hit send, and the files will silently disappear. That's a UX trap even if file upload is planned for a future iteration — a "files not yet supported" indicator or disabled send would be less confusing.

The "Scan" option (`_pickScan`) is identical to `_pickCamera` — same `ImageSource.camera` call, same `SelectedFileType.image` result. If document scanning (OCR) is the intent, there's no scanning logic present; it's a duplicate of Camera with a different label.

## Artifact detection heuristics

`ArtifactViewer.detectArtifactType` runs on every AI message's content. The detection cascade:

1. If content contains `'```html'` → HTML artifact
2. If content contains any of a hardcoded list of language fences → code artifact
3. If content is > 200 chars and has >= 2 lines starting with `#` → markdown artifact
4. Otherwise → unknown (no button shown)

The false-positive path: a message like "Here's how to format Dart code: use \`\`\`dart at the start" triggers the artifact button and opens a viewer showing the text portion as extracted content. Anchoring the regex to line boundaries (`\n\`\`\`\w+\n`) would eliminate most false positives.

The `extractArtifactContent` method uses `RegExp(r'```\w*\n([\s\S]*?)```')` with a non-greedy match, so only the first code block is extracted even if multiple exist. Multi-block messages will only preview the first block in the full-screen viewer, which may confuse users who expected to see the entire response.

## Prompt maker service instantiation

```dart
final improvedPrompt = await ChatService().makePrompt(
  roughText: currentText,
  provider: widget.selectedProvider,
  model: widget.selectedModel,
);
```

Every tap allocates a new `ChatService` → new `ApiService` → new `Dio` instance → new interceptor chain. There's no connection reuse, no shared cookie jar, and each abandoned instance lingers until GC. The rest of the app accesses `ChatService` through Riverpod's `chatRepositoryProvider`. The prompt maker should accept the service as a constructor parameter or read it from a provider.

## Dead `onFileRemoved` parameter

`FileUploadWidget` declares `final Function(int index) onFileRemoved` as a required constructor parameter. The widget never calls this callback — it doesn't track selected files internally and doesn't render removal UI. The parent wires it up (`onFileRemoved: (int index) { setState(() { _selectedFiles.removeAt(index); }); }`) but that code path is never triggered from within the widget. The actual removal happens through the preview bar's close button, which directly mutates `_selectedFiles`. This is harmless but misleading — the parameter suggests the widget manages removal state when it doesn't.
