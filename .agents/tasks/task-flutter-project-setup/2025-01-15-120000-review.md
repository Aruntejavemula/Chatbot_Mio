# Flutter project scaffolding for Mio chatbot app

This change bootstraps a full Flutter project at `frontend/` with org `com.mioapp` and project name `mio`. It adds 24 production dependencies and 4 dev dependencies covering state management (Riverpod), routing (GoRouter), networking (Dio), auth (Supabase, Google, Apple), payments (Razorpay, Stripe), and UI utilities. The `lib/` directory is pre-populated with 45 placeholder files organized into a three-layer architecture (core, data, presentation). Platform scaffolding for Android, iOS, macOS, Linux, Windows, and web is included via Flutter's default templates.

**Watch for:** No providers directory exists despite Riverpod being the chosen state management approach (confirmed gap in folder structure). Two dependency package names were substituted from the original spec (likely fine, but undocumented outside task metadata). The `main.dart` is still the default counter demo — it should be replaced with the actual app entry point using Riverpod and GoRouter before any feature work begins.

## High-level view

The project skeleton follows a clean architecture pattern with `core/` for cross-cutting concerns (theme, constants, utilities), `data/` for models, services, and repositories, and `presentation/` for screens and widgets. The folder breakdown maps well to the app's domain: auth flows, chat, settings, and payment. No providers directory exists yet despite Riverpod being the chosen state management solution — the next developer has no guidance on where state notifiers and providers should live.

Dependencies are aggressive for a scaffolding commit — 24 packages resolved at once. Two packages from the original spec (`stripe_flutter`, `code_highlight`) don't exist on pub.dev and were substituted with `flutter_stripe` and `highlight_text` respectively. The `pubspec.lock` is committed, pinning all transitive dependencies for reproducible builds.

The `main.dart` uses Dart 3.12's static access shorthand syntax (`.fromSeed(...)`, `.center`) which is valid for the declared SDK constraint (`^3.12.0`). However, it remains the default Flutter counter demo with no Riverpod `ProviderScope`, no GoRouter setup, and no reference to any of the declared dependencies — meaning the scaffolding doesn't validate that the dependency graph actually compiles in the context of the intended architecture.

## Missing state management layer

The pubspec declares `flutter_riverpod`, `riverpod_annotation`, and `riverpod_generator` as dependencies, but the folder structure has no home for providers or state notifiers. The typical Riverpod project includes a `providers/` directory (or co-locates providers with features). Without this, the next developer implementing actual logic has no guidance on where state management code should live. A `presentation/providers/` or top-level `providers/` directory with placeholders for `auth_provider.dart`, `chat_provider.dart`, and `settings_provider.dart` would complete the architecture.

Similarly, there's no placeholder for a router configuration file despite `go_router` being a core dependency. A `core/router/` or `presentation/router/app_router.dart` placeholder would signal intent.

## Dependency substitutions and the encrypt gap

The task required `stripe_flutter: ^10.1.0` and `code_highlight: ^0.4.0`, neither of which exist on pub.dev. The implementation substituted `flutter_stripe: ^11.3.0` and `highlight_text: ^1.6.1`. These substitutions are documented in the task JSON but not in `pubspec.yaml` comments or any developer-facing documentation.

The `encrypt` package (`^5.0.3`) is listed alongside `crypto: ^3.0.3`, but the folder structure has no `crypto_service.dart` or `encryption_util.dart` placeholder. If encryption is planned (likely, given the secure storage and API key management features), the architecture should signal where that logic will live.

## Platform configuration — implicit minSdk

The Android `build.gradle.kts` uses `flutter.minSdkVersion` for `minSdk`, which defaults to 21 in Flutter 3.44. Several dependencies (`flutter_secure_storage`, `flutter_stripe`, `razorpay_flutter`) require minSdk 21 or higher, so this works today — but it's implicit. If a future Flutter release changes the default, builds could break. Pinning `minSdk = 21` (or higher, since `flutter_stripe` may need 23+) would be defensive.
