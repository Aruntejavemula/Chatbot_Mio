# Responsive layout, Projects CRUD, and premium animations

This change adds three capabilities in a single batch: adaptive layout that switches between phone, tablet, and desktop modes via a `Responsive` utility; a full Projects feature (Supabase schema, FastAPI CRUD router gated to Pro users, and a Flutter frontend with model/service/screens/sidebar section); and an `MioAnimations` utility class that drives slide+fade transitions on route changes, chat bubbles, sidebar, loading dots, and settings tiles. The backend router is scaffold-only — every endpoint returns placeholder data or raises 404 because no Supabase client is wired yet. The frontend Projects section is likewise display-scaffolding wired to an empty provider and a service that will 404 against the current backend.

Watch for: the `update_project` endpoint is unreachable due to always-None placeholder logic (confirmed); the `_check_pro_plan` gate reads a `plan` field that the auth stub never provides, so all authenticated users get 403'd in practice (confirmed); the sidebar's `CreateProjectSheet` `onCreated` callback is a no-op — projects created by the user are silently discarded (confirmed).

## High-level view

The responsive strategy uses `MediaQuery` width breakpoints (600 / 1100) in a static `Responsive` class. `ChatScreen` switches between a permanent sidebar `Row` layout and an overlay drawer depending on the breakpoint. `ChatListScreen` adds a split-panel mode. The approach introduces a second sidebar instance (permanent vs overlay) rather than one sidebar with two presentation modes, which means sidebar state (search, selection) is not shared between them.

The Projects backend defines five endpoints under `/projects` — create, list, update, delete, list-chats — all gated by `get_current_user` and `rate_limit` and a `_check_pro_plan` helper. The placeholder implementation makes every mutable endpoint hit the `project_data is None` check and 404, and the list endpoints return empty arrays. The plan check reads `current_user.get("plan")`, but the auth middleware only returns `{"user_id": token[:8]}`, so every request will fail the Pro gate regardless — the router can't be smoke-tested.

The Projects frontend provides `ProjectModel`, `ProjectService`, `ProjectScreen`, `CreateProjectSheet`, and a `ProjectListScreen` stub. A `projectsProvider` in the sidebar holds local state but is never populated from the service. `CreateProjectSheet` accepts a callback but the sidebar passes a no-op, so the creation form submits to nowhere.

Animations are centralised in `MioAnimations` (durations, curve, spring constants). Route transitions use `CustomTransitionPage` with a right-to-left slide applied globally — including auth flows where the directional metaphor breaks. Chat bubbles animate in with slide+fade via a per-bubble `AnimationController`, which means re-animation on scroll-back in long conversations.

## Backend endpoints are scaffold-only with dead paths

The placeholder implementation creates unreachable code paths. `update_project` and `delete_project` both set `project_data = None` then immediately check `if project_data is None: raise HTTPException(404)`. The ownership check, update-field assembly, and response construction after it are dead code that can't be exercised even in manual testing. `create_project` sidesteps this by returning a hardcoded response, and `list_projects` / `list_project_chats` return empty lists, so those at least respond successfully.

`_check_pro_plan` reads `current_user.get("plan", "free")`. The auth middleware at `app/middleware/auth.py` returns only `{"user_id": token[:8]}` — no `plan` key ever. The default `"free"` always wins and every request is 403'd. The router can't be smoke-tested end-to-end even with the placeholder data, which defeats the purpose of the scaffold.

## Frontend Projects wiring gap

`projectsProvider` is a `StateProvider<List<ProjectModel>>` initialized to `[]` and never updated. The sidebar reads from it, so the PROJECTS section always renders empty (though the section itself always shows because `isProPlan` is `true`).

`CreateProjectSheet` receives an `onCreated` callback from the sidebar:

```dart
onCreated: (name, color, systemPrompt) {
  // Placeholder: would call service and refresh
},
```

The user fills out the form, taps Create, the sheet pops, and nothing happens. No error, no feedback, no data persisted. This is the kind of silent failure that's confusing during QA — either wire it or show a disabled state.

## Color parsing and validation

The frontend's `_parseProjectColor` splits on `#` and parses 6-character hex. If the server stores a value that's not exactly 6 hex chars (e.g., 3-char shorthand, or garbage from an unvalidated input), parsing will throw `FormatException` from `int.parse` or silently fall through to the default. The backend validates `max_length=7` (correct for `#RRGGBB`) but doesn't enforce the format — a string like `"notahex"` passes Pydantic validation and will be stored.

## Responsive layout dual-sidebar

`ChatScreen` instantiates `SidebarWidget` in two places: once as a permanent sidebar (when `showPermanentSidebar` is true) and once as an overlay (when `_isSidebarOpen` is true and not in permanent mode). These are separate widget instances, so search query state (`_searchQuery`) in one doesn't carry to the other. Searching in one mode is lost when rotating the device or resizing the window.

## Route transition direction mismatch

Every route — including splash, welcome, login — now uses the same slide-from-right `CustomTransitionPage`. Navigating *back* to splash (e.g., after sign-out) plays a forward-slide that feels directionally wrong since it's a "reset" navigation, not a "deeper" navigation. Auth flows and reset navigations would benefit from a fade or no-transition page.

## projectNewChat loses project context

The route `/projects/:projectId/new-chat` is defined as:

```dart
GoRoute(
  path: AppRoutes.projectNewChat,
  pageBuilder: (context, state) => _buildTransitionPage(
    key: state.pageKey,
    child: const ChatScreen(chatId: null),
  ),
),
```

The `projectId` path parameter is extracted nowhere. `ChatScreen` receives no project reference, so the project's `system_prompt` — the main value proposition of projects — can't be injected into the new chat session. The route currently functions identically to navigating to `/chat`.
