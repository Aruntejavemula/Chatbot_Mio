# Test infrastructure across frontend and backend

Two branches deliver test scaffolding for the entire monorepo. The frontend branch (`feature/project-setup-resolved`) adds 71 Dart tests (unit + widget) with mocktail, fixing pre-existing compile and dependency issues along the way. The backend branch (`feature/backend-setup`) introduces a new `backend/` directory structure with FastAPI services, 70 Python tests (unit/integration/security/e2e), and two GitHub Actions workflows. The approach is test-first: new backend services (`CostService`, `EncryptionService`, `CalculatorSkill`, `WebScraperSkill`, `SkillRegistry`) were created specifically as test targets rather than extracted from existing production code.

**Watch for:** The encryption service uses a hand-rolled XOR cipher that is not cryptographically secure (confirmed). The auth middleware validates only token length, making any 10+ character string a valid credential (confirmed). The "e2e" tests are actually integration tests hitting the ASGI transport, not browser-based Playwright tests despite Playwright being listed as a dependency (confirmed). CI workflows never run `flutter test`, only `flutter analyze` (confirmed).

## High-level view

The frontend tests cover model serialization and widget behavior with meaningful assertions. The widget tests handle Flutter's timer lifecycle correctly — a common source of flakiness — though the `TrialBannerWidget` has zero behavioral coverage due to its `FlutterSecureStorage` dependency throwing in test environments.

The backend introduces production-like service code whose sole purpose is to give the tests something to exercise. The `EncryptionService` is a cryptographic footgun: XOR with a single SHA-256 pass provides no real confidentiality guarantee and must not ship to production.

The auth layer uses token-length as its only validation heuristic. Every endpoint depends on `get_current_user`, which accepts any string of 10+ characters as a valid Bearer token and derives `user_id` from the first 8 characters. The rate limiter keys on this fake user_id, so the entire auth/rate-limit stack is a functional skeleton without real security properties. The `_check_pro_plan` helper reads a `plan` field that auth never sets, so the happy path (successful project creation) is never tested.

The CI/CD workflows enforce 70% coverage and ruff linting, but neither `ci.yml` nor `pr-check.yml` runs `flutter test` — only `flutter analyze`. The 71 frontend tests are never executed in CI.

7. **CI never runs frontend tests** — Both `ci.yml` and `pr-check.yml` run `flutter analyze` only, not `flutter test`. The 71 frontend tests have no CI execution. (confirmed)

## EncryptionService: XOR cipher labeled as AES-256

The docstring says "AES-256-compatible XOR cipher" and the class-level note acknowledges it's "simplified for demonstration," but the code has no relationship to AES. It performs `SHA-256(key + salt)` to produce a 32-byte key stream, then XORs the plaintext against that stream (repeated to cover the message length).

```python
# backend/app/services/encryption_service.py line 22-24
salt = os.urandom(16)
key_stream = hashlib.sha256(self._key + salt).digest()
extended_key = key_stream * ((len(data) // len(key_stream)) + 1)
```

For messages longer than 32 bytes, `extended_key` is the same 32 bytes repeated. Byte 0 and byte 32 of the plaintext are XORed with the same key byte — a classic stream cipher failure mode. No test checks that two encryptions of the same plaintext produce different ciphertexts (they do, because of the random salt, but this semantic property should be asserted).

## Auth middleware and the untestable happy path

The auth dependency accepts any Bearer token of 10+ characters and returns `{"user_id": token[:8]}`. The rate limiter keys on this `user_id`, so rate limiting is trivially bypassed by rotating the first 8 characters.

The `_check_pro_plan` helper reads `current_user.get("plan", "free")` but auth never sets a `plan` field. Every authenticated user hits the 403 "not pro" path. The integration tests verify this behavior (which is internally consistent), but it means:
- Project creation success path is never tested
- Project update/delete ownership checks are dead code (the `project_data = None` assignment guarantees 404 before ownership is checked)

## E2E test mislabeling

`backend/tests/e2e/test_web_app.py` uses `httpx.AsyncClient` with `ASGITransport` — the exact same pattern as the integration tests. The `conftest.py` sets up a `base_url` fixture for Playwright but it's never imported or used. The `pytest.ini` marks these as `e2e` and CI excludes them with `-m "not e2e"`, creating a false sense of e2e coverage while the actual tests provide no additional signal beyond what integration tests already cover.

## Rate limiter test coupling

Every integration and security test file includes:
```python
@pytest.fixture(autouse=True)
def reset_rate_limiter():
    _rate_limiter._requests.clear()
    yield
    _rate_limiter._requests.clear()
```

This reaches into the private `_requests` dict of the module-level singleton. A `reset()` method on `RateLimiter` or dependency injection of the limiter instance would decouple tests from the implementation.

## CI gap: frontend tests never run

Both CI workflows include Flutter jobs that run `flutter pub get` and `flutter analyze` but never `flutter test`. Adding a `flutter test` step is trivial and would catch regressions in the 71 tests this PR introduces.
