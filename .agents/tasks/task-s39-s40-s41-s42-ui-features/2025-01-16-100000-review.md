# Export, usage dashboard, cloud sync, and token cap UI

Four features added to the Mio Flutter app: chat export (markdown/PDF/bulk via share sheet), a usage dashboard with token cap cards and a 7-day bar chart, a cloud storage sync screen with plan-gated provider selection, and a token cap warning banner in the chat view. The approach is UI-first — all backend endpoints are assumed to exist but none are implemented in this diff. The sync service is entirely stubbed (empty method bodies for Drive/iCloud/OneDrive). The export and usage features call real HTTP endpoints through the existing `ChatService` + `ApiService` layer.

**Watch for:** Token cap banner displays whenever `tokenCapProvider` is non-null regardless of percentage (confirmed), export temp files are never cleaned up (confirmed), sync service stores chat data in `FlutterSecureStorage` which has low size limits (likely), usage screen timer fires even when the screen is backgrounded (likely).

## High-level view

The export feature adds three service methods that fetch binary blobs from the backend, write them to temp files, and share via `share_plus`. The bottom sheet gates PDF export behind paid plans and bulk export behind pro, but the "Upgrade" button in the dialog doesn't actually navigate to the subscription screen.

The usage dashboard polls `GET /tokens/usage` every 60 seconds via `Timer.periodic`, renders four token cap progress cards, a cost transparency section, and a basic bar chart. The timer runs unconditionally while the screen's state object is alive, including when the app is backgrounded.

The cloud storage screen presents a plan-based UI (free / basic / pro) but has no connection to the `SyncService` — selecting a provider updates local state only without persisting or triggering any sync. The `SyncService` itself has empty cloud methods and uses `FlutterSecureStorage` for local persistence, which is designed for small secrets, not serialized chat histories.

The token cap banner integrates into the chat screen by watching a Riverpod `StateProvider` that gets populated after each streamed response. The banner handles three states (hidden / warning / blocked), but the chat screen renders it whenever the provider is non-null, so a stale warning from a previous message remains visible even if usage has since dropped below 70%.

## Token cap banner lifecycle

`tokenCapProvider` is set to non-null inside `ChatRepository.sendMessage` after the stream completes, if a `cap_warning` event arrived in the SSE stream. It is never reset back to null — not at the start of the next `sendMessage`, not on screen navigation, not on chat switch.

```dart
if (tokenCap != null)
  TokenCapBanner(
    capType: (tokenCap['cap_type'] as String?) ?? '',
    used: (tokenCap['used'] as int?) ?? 0,
    limit: (tokenCap['limit'] as int?) ?? 1,
    ...
  ),
```

The `TokenCapBanner` widget returns `SizedBox.shrink()` when `percentage < 0.7`, which provides a partial safety net. But the common case — stale data set at 75%, remains at 75% indefinitely — isn't handled. The `lastCapWarning` field on `ChatService` is reset to null at the start of `streamMessage`, but `tokenCapProvider` in the repository is only written to when `lastCapWarning != null`. A subsequent message without a cap_warning event leaves the old value in the provider.

## Export file handling and access gating

The three export methods write to fixed temp paths (`chat_export.md`, `chat_export.pdf`, `all_chats_export.md`) and never clean up. The `share_plus` API doesn't notify when the user finishes the share sheet, so cleanup timing is inherently awkward — but a `finally` block that deletes the file after awaiting the share call would handle the common case.

The upgrade dialog is a dead end:

```dart
TextButton(
  onPressed: () => Navigator.of(dialogContext).pop(),
  child: Text('Upgrade', ...),
),
```

This should call `context.go(AppRoutes.subscription)` after popping.

The `chatId` passed to the export sheet is `widget.chatId ?? ''`. The export button is only visible when `messages.isNotEmpty`, which partially mitigates this, but `widget.chatId` can still be null if messages were sent in-memory without creating a backend chat first — resulting in a request to `/export/chat//markdown`.

## Usage dashboard polling

On mobile the screen's state object remains alive while the app is in the background (the route is still mounted in the navigation stack). The timer continues firing `_loadUsage` every 60 seconds, making network requests the user won't see. Adding `WidgetsBindingObserver` and pausing/resuming the timer on lifecycle changes is the standard fix.

## Sync service architecture

`SyncService` is entirely disconnected from the rest of the app. Nothing in `StorageScreen` instantiates or calls it. The `_loadStorageSettings` method in `StorageScreen` is a no-op that hardcodes `free` plan and null provider.

The `SyncService.saveLocally` implementation uses `FlutterSecureStorage` — backed by the platform keychain/keystore — to persist serialized JSON chat objects. On iOS the Keychain becomes slow with large values; on Android, `EncryptedSharedPreferences` stores everything in a single file fully loaded into memory. A chat with thousands of messages serialized to JSON could be hundreds of KB per entry.

The `loadLocally` type check is a no-op:

```dart
final decoded = jsonDecode(jsonString);
if (decoded is Map<String, Object?>) return decoded;
return null;
```

`jsonDecode` returns `dynamic`, and `dynamic` satisfies `Object?`, so this check always passes for any map result. It provides no actual type safety.

## Platform.isIOS in StorageScreen

`_buildBasicContent` uses `Platform.isIOS || Platform.isMacOS` from `dart:io` to conditionally show the iCloud tile. On web builds, `dart:io` is unavailable and this throws at runtime. If the app targets Flutter web in the future, this needs a `kIsWeb` guard. (possible)
