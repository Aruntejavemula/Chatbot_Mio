# 8-page onboarding flow, funny warnings system, and permission dialogs

Three features land together: a complete rewrite of the onboarding screen into an 8-page PageView flow with preference persistence, a centralized warning-message utility with a custom snackbar widget, and a reusable permission dialog shown before OS permission requests. The onboarding collects user preferences (SharedPreferences) and marks completion (FlutterSecureStorage), the warnings system replaces scattered plain SnackBars with a branded penguin snackbar, and the permission dialogs gate microphone/camera/photos access behind an in-app "Allow/Deny" step.

**Watch for:** API key entered on page 6 is never persisted (confirmed). The BYOK provider grid uses `Map<String, dynamic>`, violating the project's strict-no-dynamic rule (confirmed). `TickerProviderStateMixin` is retained on the onboarding state class despite no longer having any animation controllers (confirmed). The "sign in" snackbar uses a raw string instead of a `FunnyWarnings` constant (confirmed).

## High-level view

The onboarding rewrite trades the old 5-page swiping flow for a locked 8-page `NeverScrollableScrollPhysics` PageView driven entirely by buttons. Navigation state is duplicated — `_goToPage` calls `setState` immediately while `onPageChanged` fires redundantly during the same animation — but in practice this is harmless since the physics are disabled and no external swipe can trigger `onPageChanged` independently.

The funny warnings system is small and well-contained: a `FunnyWarnings` class of `static const String` fields plus a `FunnySnackbar.show()` that wraps `ScaffoldMessenger` with a branded floating container. One usage inconsistency exists: the "sign in" message in `chat_screen.dart` bypasses the constants.

The permission dialog sits between user intent and OS permission prompts. It always shows — there is no check for "already granted" status, meaning returning users see the dialog every time they tap record or attach a photo, even if the OS already granted the permission.

## API key input page collects but never stores

Page 6 presents a `TextField` bound to `_apiKeyController` with an obscured input and visibility toggle. The "Next" button calls `_nextPage()`, which simply advances to page 7 without reading `_apiKeyController.text`. When onboarding completes, only `onboarding_complete` is written to `FlutterSecureStorage`. The user's entered key is silently discarded.

```dart
void _nextPage() {
  if (_currentPage == 4) {
    _savePreferences();  // saves chips, not key
  }
  if (_currentPage < _totalPages - 1) {
    _goToPage(_currentPage + 1);   // no key save on page 5->6 transition
  } else {
    _completeOnboarding();  // only writes 'onboarding_complete'
  }
}
```

This is a UX and functional gap: users who paste a key and continue will believe it was saved.

## Dynamic types in the BYOK provider grid

The project convention mandates strict null safety with no `dynamic` types. The provider grid on page 4 declares:

```dart
final providers = <Map<String, dynamic>>[
  {'label': 'O', 'name': 'OpenAI', 'color': const Color(0xFF10A37F)},
  ...
];
```

Every access site does `provider['color'] as Color` — a runtime cast that masks type errors. A simple data class (`_ProviderInfo` with `String label`, `String name`, `Color color`) would eliminate the dynamic map and the casts.

## Dead TickerProviderStateMixin

The original onboarding had a `_floatingController` animation that justified `TickerProviderStateMixin`. The rewrite removed that animation entirely (replaced by `PenguinMascot` which provides its own ticker internally). The mixin remains on line 21 with no consumers. It adds unnecessary overhead (multiple `Ticker` allocation paths) and confuses future readers about where animations live.

## Inconsistency in FunnySnackbar adoption

`chat_screen.dart` calls `FunnySnackbar.show(context, 'Please sign in to chat', ...)` with a hardcoded string. The second call on the same screen correctly uses `FunnyWarnings.modelNotSelected`. This defeats the purpose of centralized warnings — if the sign-in message changes, it requires a code search instead of a single constant update.

## Permission dialog fires regardless of current grant state

`PermissionDialog.microphone(context)` is called unconditionally in `_startRecording()`. On iOS/Android, once a permission is granted it stays granted until revoked. Showing the dialog every recording session creates friction for returning users. The standard pattern:

```dart
final status = await Permission.microphone.status;
if (status.isGranted) { /* proceed */ }
else { /* show dialog, then request */ }
```

The same applies to `PermissionDialog.photos` and `PermissionDialog.camera` in `plus_panel_widget.dart`.

## Mixed opacity APIs within a single PR

`onboarding_screen.dart` uses the newer `withValues(alpha: 0.1)` pattern (line 691), while `funny_snackbar.dart` and `plus_panel_widget.dart` use the deprecated `withOpacity(0.08)` / `withOpacity(0.1)`. Within a single PR the convention should be consistent. `withOpacity` is deprecated in Flutter 3.27+; the entire PR should use `withValues(alpha:)`.
