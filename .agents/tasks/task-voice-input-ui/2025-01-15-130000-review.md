# Voice input with Whisper transcription backend

This change adds end-to-end voice input to the Mio chat app. The Flutter frontend gains a `VoiceInputWidget` that records audio via the `record` package, cycles through idle/recording/processing/error states with a waveform animation, and sends the file to a new FastAPI backend at `POST /voice/transcribe`. The backend validates file format and size, forwards to OpenAI Whisper, and returns the transcript. Auth (JWT placeholder) and rate limiting (30 req/min in-memory sliding window) are wired as FastAPI dependencies.

**Watch for:** VoiceService bypasses the existing ApiService and its auth interceptor, so requests to `/voice/transcribe` will arrive without the JWT token the backend requires (confirmed). The in-memory rate limiter leaks entries indefinitely since there's no eviction of keys whose timestamp lists are empty (confirmed). CORS is `allow_origins=["*"]` with `allow_credentials=True`, which FastAPI translates to echoing any request origin—an open relay (confirmed).

## High-level view

The Flutter widget is a state machine with four visual states driven by a `_VoiceInputState` enum. The main integration gap is that it instantiates its own `VoiceService` (a standalone Dio client) rather than going through `ApiService`, which means the stored JWT is never attached to transcription requests. Every call to `/voice/transcribe` will 422 on the missing `Authorization` header.

The backend is a from-scratch FastAPI app with a single router. The auth dependency is a placeholder that accepts any string longer than 10 characters as a valid token, and the rate limiter keys on client IP rather than authenticated user identity, so all users behind a shared IP (common on mobile networks) share one 30-request bucket.

The rate limiter's `_requests` dict grows without bound. Expired timestamps within a key get pruned on access, but keys themselves are never removed once a client stops sending requests. Under sustained multi-client load this is a memory leak proportional to the number of distinct IPs ever seen.

1. **VoiceService skips auth token** — `VoiceService` creates its own Dio instance without the auth interceptor, so the `Authorization` header required by `get_current_user` is never sent. Route it through `ApiService` or replicate the token-attach logic.
2. **Rate limiter memory leak** — `_requests` dict accumulates keys forever; add periodic eviction of empty/stale keys or switch to a bounded data structure.
3. **Rate limit keys on IP, not user** — Multiple users behind a single NAT share one rate-limit bucket; restructure the dependency to accept the auth result and key on user_id.
4. **CORS wildcard with credentials** — FastAPI echoes the request origin when credentials are enabled with `*`, making it "allow all origins with credentials." Replace `*` with the actual frontend origin before any deployment.
5. **Hardcoded recording path** — `/tmp/recording.m4a` means concurrent widget instances or rapid re-mounts would race on the same file; use a unique temp file per session.
6. **Orphaned recording on navigation** — If the user navigates away mid-recording, `dispose()` fires but the in-progress recording file is never cleaned up or uploaded.

## VoiceService bypasses ApiService auth

The existing `ApiService` class attaches the JWT from `FlutterSecureStorage` via a Dio interceptor on every request. `VoiceService` creates a fresh `Dio()` instance with no interceptors:

```dart
VoiceService()
    : _dio = Dio(
        BaseOptions(
          baseUrl: _baseUrl,
          connectTimeout: const Duration(seconds: 60),
          receiveTimeout: const Duration(seconds: 60),
        ),
      );
```

The backend's `get_current_user` dependency requires `Authorization: Bearer <token>` (header is declared with `Header(...)`—no default, so missing header = 422). Every transcription request from the Flutter app will fail with a validation error before reaching any business logic.

Fix: either inject `ApiService.dio` into `VoiceService`, or have `VoiceService` read the token from secure storage and attach it. The second approach duplicates logic; the first is cleaner.

## Rate limiter unbounded growth

`RateLimiter._requests` is a `dict[str, list[float]]`. When `_cleanup_expired` runs, it filters timestamps within a key but never deletes the key itself when the list becomes empty. Over time, every distinct client IP that ever hit the endpoint stays in memory as an empty list.

A simple mitigation: delete the key when the list is empty after cleanup:

```python
if not self._requests[key]:
    del self._requests[key]
```

## Rate limit identity mismatch

The rate limiter's dependency function keys on client IP:

```python
client_ip = request.client.host if request.client else "unknown"
key = f"rate_limit:{client_ip}"
```

The comment says "prefer user_id if available," but the dependency has no access to the `current_user` result because FastAPI dependencies execute independently unless one depends on the other. Since `get_current_user` already runs on the same endpoint, the user_id could be passed through a sub-dependency or by restructuring the rate-limit dependency to accept the auth result as a parameter.

## Hardcoded recording path and orphaned files

```dart
static const String _recordingPath = '/tmp/recording.m4a';
```

On iOS `/tmp` is sandboxed, so cross-app collision isn't the concern—but if the widget is mounted twice (split-screen, rapid state transitions) two instances race on the same file. Using `path_provider`'s `getTemporaryDirectory()` plus a timestamp suffix would eliminate this.

Additionally, if the user navigates away mid-recording, `dispose()` cancels the timer and disposes the controller, but the recording file at that path is never deleted. The orphaned file persists until the OS reclaims temp storage.
