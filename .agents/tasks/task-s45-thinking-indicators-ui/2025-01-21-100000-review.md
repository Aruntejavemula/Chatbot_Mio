# Thinking indicators UI for streaming chat

This change adds a collapsible "thinking" block to the chat interface, allowing users to see model reasoning (chain-of-thought) content as it streams in. The implementation introduces a typed event system (`ChatStreamEvent`) to distinguish thinking chunks from text chunks in the SSE stream, adds new Riverpod state providers for thinking text, and renders a `ThinkingBlockWidget` with an animated spinner during active streaming that collapses into a disclosure triangle once complete. The thinking content is persisted on the `MessageModel` so historical messages can re-render their reasoning blocks.

Watch for: The `ChatBubble` widget receives new `thinkingContent`/`isThinkingStreaming` props but is never instantiated anywhere — dead code that will drift (confirmed). The `copyWith` pattern on `MessageModel` cannot clear `thinkingContent` back to `null` once set (confirmed, latent). State cleanup in the error path duplicates the happy path with no extraction into a helper, making future divergence likely (confirmed).

## High-level view

The streaming pipeline shifts from `Stream<String>` to `Stream<ChatStreamEvent>`, a tagged union of `thinking`, `text`, and `done` variants. The SSE parser handles both the new typed format (JSON with a `type` field) and a legacy fallback (JSON with only a `content` field), so the client is backward-compatible with servers that don't emit thinking events.

State management adds two providers (`streamingThinkingTextProvider`, `isThinkingStreamingProvider`) that mirror the existing `streamingTextProvider` pattern. The repository accumulates thinking chunks into a separate buffer and flips a boolean on phase transition. The state reset logic is duplicated across the success and error paths — a future provider addition will likely diverge.

The `ThinkingBlockWidget` renders in the chat screen's `_buildMessagesList` for both persisted and streaming messages. It also renders inside `ChatBubble`, but `ChatBubble` is not instantiated anywhere in the app.

1. **Dead ChatBubble integration** — `ChatBubble` has `thinkingContent` and `isThinkingStreaming` props added but is never instantiated. Either wire it up or remove the additions to avoid maintenance drift.
2. **copyWith cannot null thinkingContent** — `MessageModel.copyWith` uses `thinkingContent ?? this.thinkingContent`, so once set, thinking content cannot be explicitly cleared to `null`. Use a sentinel or `Optional`-style wrapper if clearing is ever needed.
3. **State reset duplication** — The four-line state reset block (`isStreaming`, `streamingText`, `streamingThinkingText`, `isThinkingStreaming`) is written identically in both the `try` body and `catch` block. Extract a `_resetStreamingState()` helper to keep them in sync.
4. **Unbounded thinking buffer** — `thinkingBuffer` accumulates all thinking tokens without any size guard. A pathological response could grow unbounded in memory before the stream completes.

## Typed SSE event parsing and fail-open for unknown types

The `type` field is used for `cap_warning` events (handled by an earlier branch), then `thinking`, `text`, and `done`. If a new event type is added server-side that doesn't match this chain, it silently falls through to the `json.containsKey('content')` fallback, which could misinterpret the payload as a text event:

```dart
if (json.containsKey('type')) {
  final eventType = json['type'] as String;
  if (eventType == 'thinking') { ... }
  else if (eventType == 'text') { ... }
  else if (eventType == 'done') { return; }
  // unknown types fall through silently to content fallback
}
```

This fail-open behavior is arguably correct for forward-compat but worth a `continue` after the typed-event block to prevent the fallback from firing on typed events with a `content` field.

## Phase transition assumes strict ordering

The repository flips `isThinkingStreamingProvider` to `true` on the first thinking event and back to `false` on the first text event. If the server ever interleaves thinking and text chunks, the UI state would toggle rapidly between the spinning and static icons, causing visual flickering. The current implementation assumes a single thinking phase followed by a single text phase with no interleaving.

## Dead `ChatBubble` integration

`ChatBubble` gains `thinkingContent` and `isThinkingStreaming` properties and renders a `ThinkingBlockWidget` when content is present. However, `ChatBubble` is never instantiated — `chat_screen.dart` builds message widgets inline. These props will drift from any future refactors to the inline rendering without compiler warnings.

## `copyWith` cannot clear nullable fields

`MessageModel.copyWith` uses `thinkingContent ?? this.thinkingContent`. Callers can override the value but cannot explicitly set it to `null`. Currently no code calls `copyWith` on `MessageModel`, so the issue is latent but will surface if message editing or regeneration is added.

## State cleanup duplication

```dart
_ref.read(isStreamingProvider.notifier).state = false;
_ref.read(streamingTextProvider.notifier).state = '';
_ref.read(streamingThinkingTextProvider.notifier).state = '';
_ref.read(isThinkingStreamingProvider.notifier).state = false;
```

This block appears twice (success and catch). A fifth provider will likely be added to only one of the two.
